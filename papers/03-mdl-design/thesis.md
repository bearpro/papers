## Проблема

- Я пытаюсь решить задачу поиска противоречий в нормативных документах
- Для того чтобы находить противоречия мне нужен формальный способ записи нормативных положений
- Математический аппарат уже разработан: формализм DDL-LTLf (дефизибельная деонтическая логика с темпоральной семантикой), и показа алгоритм поиска противоречий. [Проказин, ИТ-Стандарт, 2026 год]
- Подход к программной реализации в целом понятный: специальный формальном язык промежуточного представления нормативных документов, наподобие ReqIF и других, но с учётом специфики задачи
- У такого подхода есть ряд проблем, решения которых рассмотрены в данной статье:
  - Интеграция DDL-LTLf с обычным алгоритмами  
    Формализм описывает поведения деонтических требований и отношения противоречия. В процессе моделирования реальных требований этих строительных блоков недостаточно: нужны стркои, числа, функции, коллекции, модули, и другие примитивы языков программирования. При этом всё это должно разрешаться в формальную семантику DDL-LTLf.
  - Семантическая интеропрабельность  
    два совершенно разных кода на специальном языке, написаны разными людьми (или LLM), в разных контекстах и в разное время, на основе разных исходных документов. А эти коды на формальном языке должны друг с другом корректно состыковаться, при чём даже не попарно, а на больших массивах документов.  
    Решение этой проблемы требует комплексного подхода:
    - Строгая формальная семантика языка
    - Устойчивая грамматическую семантику для неформальных объектов языка (в основном имён)
    - Межмодельное выравнивание
  - Трассируемость к источнику  
    Необходимо как при работе с кодом, так и в процессе поиска противоречий иметь инструмент отслеживания источника требований.
  - LLM-дружелюбность  
    Одним из важных сценариев использования для языка является генерация кода по исходному документу. Это требует учёта в дизайне языка, для того чтобы не создавать лишних сложностей для обучения или для использования предобученных моделей, не знакомых с языком.

# Структура программы на языке

Так как основная часть грамматики и языка как системы в целом не обладает новизной, в статье она не рассматривается.
Основная цель данного раздела: познакомить читателя с синтаксисом, используемым в примерах в других разделах. Кроме того, здесь обоснования некоторых решений в дизайне языка, которые показались автору особенно удачно вдохновлены другими языками.

## ML-подобный синтаксис в качестве основы

Языки группы ML удачно решают две проблемы, актуальные для рассматриваемого языка

- Моделирование доменных объектов с помощью типов  
  ML-like языки традиционно реализуют алгебраическую систему типов. Это полезное свойство позволяет эффективно ещё до начала работы DDL-LTLf ядра языка описывать сущности, и уточнять отношения между разными объектами.
- Порядок редукции  
  Так как семантика DDL-LTLf требует чтобы вся модель сводилась к атомарным пропозициям связанным LTLf-формулами и деонтическими модальностями, такие свойства ML как
  
  - Отсутствие специальных значений как `null`
  - Один `return` на функцию

  Позволяет упростить редукцию алгоритмических конструкций общего назначения к DDL-LTLf. Фактически, все конструкции общего назначения (такие как алгебраические операции, условные операторы, работа со строками и коллекциями, и т.д.) тривиально сводятся к пропозициям.

## Общий взгляд на поведение исходного кода

Чтобы лучше понять как работает язык можно представить такую иерархию для объектов исходного кода

1. Конструкции общего назначения такие как `let max_weight = 20`
   позволяют алгоритмически описывать моделируемый объект, и не могут сами по себе вступать в темпоральные или деонтические отношения
2. Некоторые конструкции общего назначения (имеющие тип
   `bool`) эквивалентны предикатам, образующим пропозицию. Например

   ```ml
   func below_max_weight(weight) = 
     weight <= max_weight
   ```

3. Пропозиции могут вступать в темпоральные отношения с помощью LTLf операторов: `below_max_weight(weight) always`. Таким образом функции возвращающие `bool` эквивалентны классу LTLf-формул.
4. LTLf-формулы образуют деонтические правила, связываясь доентическими модальностями и другими операторами DDL-LTLf

   ```ml
   entity weight: rat
   rule O below_max_weight(weight) always
   ```

   Таким образом образуются DDL-LTLf формулы, привязанные к конкретной модели (`entity`)

## Структура программы

1. Код на языке компонуется как модули
2. Модуль состоит из объявлений
3. Объявления
   - Типы
   - Значения
   - Функции
   - Сущности
   - Деонтические правила

Таким образом каждый модуль фактически экспортирует множество типизированных сущностей с именами, и множество деонтических правил которым эти сущности должны подчиняться.

```ml
@ rfc2822
module email_address_specification

import std.system.strings as strings

private type ProcessingState = LocalPart | Domain

private func process_email(
    email: List<string>, 
    state: ProcessingState) -> bool:
    case email:
    | List.Cons(char, tail): 
        if state = LocalPart and char = "@"
        then process_email(tail, Domain)
        else if char = "@" then false
        else process_email(tail, state)
    | Empty: true

private func email_is_correct(email: string) -> bool:
    let char_list = strings.to_list(email)
    let state = LocalPart
    process_email(char_list, state)

entity email: string

@ 3.4.1. Addr-spec specification
rule O email_is_correct = email_is_correct(email) always

fact "ti@pidor"
```
